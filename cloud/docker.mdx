title: "Deploy Kohya's GUI with Docker and NVIDIA GPU Support"

Docker lets you run Kohya's GUI in an isolated, reproducible container without touching your system Python environment. The official image is published to the GitHub Container Registry and includes CUDA 12.8, PyTorch, and all required dependencies. A companion TensorBoard container is included in the `docker-compose.yaml` and runs as a separate service.

## Prerequisites: GPU support

Before running the container, you need a host environment that can pass an NVIDIA GPU into Docker. The setup differs between operating systems.

### Windows

Install all four of the following components, then make sure Docker Desktop is configured to use the WSL 2 backend:

- [Docker Desktop](https://www.docker.com/products/docker-desktop/)
- [CUDA Toolkit](https://developer.nvidia.com/cuda-downloads)
- [NVIDIA Windows Driver](https://www.nvidia.com.tw/Download/index.aspx)
- [WSL 2 integration](https://docs.docker.com/desktop/wsl/#turn-on-docker-desktop-wsl-2) in Docker Desktop

<Note>
  For official guidance on GPU support in WSL 2, see the NVIDIA documentation at `https://docs.nvidia.com/cuda/wsl-user-guide/index.html` and the Docker Desktop WSL GPU docs at `https://docs.docker.com/desktop/wsl/use-wsl/#gpu-support`.
</Note>

### Linux and macOS

Install an NVIDIA GPU driver if one is not already present, then install the NVIDIA Container Toolkit:

```bash
# Install NVIDIA Container Toolkit (Debian/Ubuntu example)
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
sudo apt-get update && sudo apt-get install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
```

For the full installation guide, see `https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/latest/install-guide.html`.

## Running the pre-built image

The recommended approach is to pull the pre-built image from `ghcr.io/bmaltais/kohya-ss-gui:latest`. No local build step is required.

```bash
git clone --recursive https://github.com/bmaltais/kohya_ss.git
cd kohya_ss
docker compose up -d
```

Docker Compose will pull the image and start both the `kohya-ss-gui` service on port **7860** and the `tensorboard` service on port **6006**. Open your browser and navigate to:

- Kohya's GUI: `http://localhost:7860`
- TensorBoard: `http://localhost:6006`

### Updating the pre-built image

Pull the latest image and restart the containers:

```bash
docker compose down && docker compose up -d --pull always
```

## Building the image locally

If you want to build from source — for example, to test local changes — pass `--build` to Compose. Building from scratch can take up to 20 minutes.

<Warning>
  You must clone the repository **recursively** to include the `sd-scripts` submodule. Without it, the build will fail because the submodule source files are required at build time.
</Warning>

```bash
git clone --recursive https://github.com/bmaltais/kohya_ss.git
cd kohya_ss
docker compose up -d --build
```

### Updating a local build

Check out the new code version, then rebuild and restart:

```bash
docker compose down && docker compose up -d --build --pull always
```

## docker-compose.yaml overview

The `docker-compose.yaml` defines two services:

```yaml
services:
  kohya-ss-gui:
    container_name: kohya-ss-gui
    image: ghcr.io/bmaltais/kohya-ss-gui:latest
    ports:
      - 7860:7860
    volumes:
      - ./models:/app/models
      - ./dataset:/dataset
      - ./dataset/logs:/app/logs
      - ./dataset/outputs:/app/outputs
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              capabilities: [gpu]
              device_ids: ["all"]

  tensorboard:
    container_name: tensorboard
    image: tensorflow/tensorflow:latest-gpu
    ports:
      - ${TENSORBOARD_PORT:-6006}:6006
    volumes:
      - ./dataset/logs:/app/logs
    command: tensorboard --logdir=/app/logs --bind_all
```

## Design notes

Understanding how the Dockerfile and Compose file are structured helps avoid common mistakes.

**Dataset directory**
All training data must be stored in the `dataset` subdirectory of the cloned repository. This directory is mounted into the container at `/dataset`. Subdirectories are mapped as follows:

| Host path | Container path | Purpose |
|---|---|---|
| `./dataset` | `/dataset` | Root dataset mount |
| `./dataset/images` | `/app/data` | Training images |
| `./dataset/logs` | `/app/logs` | TensorBoard logs |
| `./dataset/outputs` | `/app/outputs` | Saved checkpoints |
| `./dataset/regularization` | `/app/regularization` | Regularization images |
| `./models` | `/app/models` | Base models |

**No file picker**
The file picker component is not available inside the container. You must type folder paths and configuration file paths directly into the input fields.

**TensorBoard is a separate container**
TensorBoard is not bundled inside the Kohya's GUI image. The `tensorboard` service in `docker-compose.yaml` runs as its own container using the official TensorFlow GPU image. The **Start TensorBoard** button in the GUI is hidden when running in Docker. Access TensorBoard directly at `http://localhost:6006`.

**No automatic browser launch**
The container runs headlessly. Open `http://localhost:7860` manually after the container starts.

**Disposable by design**
The container is designed to be discarded and recreated freely. Persistent data lives in the bind-mounted host directories, so no state is lost when you remove and recreate the container.
